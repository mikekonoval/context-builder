# architecture.md — stages for context-builder skill

Stages that populate `context/architecture/`. Stages 1-12 normally execute in sequence as part of the skill's main flow. Stage 10 (premortem) dispatches a subagent via Task tool for isolated context. Stage 13 is post-build:

- Stage 13: wikilinks graph across files

Stage 5 (memory layers) is conditional — skip if the product has no AI agent / LLM component.

## Stage 2: system map + boundaries

Read `context/architecture/01-обзор/пресс-релиз.md` and all of `context/idea/`.

Create two files in `context/architecture/01-обзор/`: `карта-системы.md` and `границы.md`.

### карта-системы.md

Mermaid diagram of all components, grouped by layers relevant to the project. Possible layers (use only those actually applicable):
- **Внешний мир** (юзеры, внешние сервисы)
- **Интерфейс** (через что юзер общается с продуктом)
- **Ядро** (наша логика)
- **Хранение** (база, кэш, файлы)
- **Интеграции** (telegram, email, calendar и т.п.)

Если слой не нужен — не выдумывай. Стрелки только осмысленные.

Если имена компонентов короткие или неочевидные — под диаграммой список с описаниями.

Также зафиксируй UX-стиль: тихий бэкенд (результаты приходят в Telegram/email/чат) или визуальный интерфейс (свой UI с анимациями, прогрессом, звуками). Для b2b-инструментов обычно тихий; для клиентских и маркетинговых — визуальный.

### границы.md

```
# Границы

## Что внутри системы (наша ответственность)
## Что снаружи (используем как сервис, доверяем)
## Граница доверия
- Что приходит от пользователя (опасно — может быть атакой)
- Что от внешних сервисов (нормально, но может упасть)
- Что от нас (полностью доверяем)
```

Important: один сервер, одна база, один деплой — пока физически не невозможно. Не выдумывай микросервисы. Не предлагай Kubernetes на 10 пользователей.

If any data is missing — ask a clarifying question.

## Stage 4: main brick (data abstraction hypothesis)

Read `context/idea/` and `context/architecture/01-обзор/`.

Create `context/architecture/03-данные/` with `главный-кирпич.md`.

**Action:**

1. Составь список всех типов сущностей из `context/idea/`.

2. Проверь гипотезу: можно ли свести их к 1-2 базовым кирпичам?

   Примеры:
   - Notion свёл всё к одному «блоку»: текст, картинка, страница, база — это типы одного объекта.
   - Linear — к «задаче».
   - Stripe — к «объекту» с типизированным id.

   Important: единая абстракция работает не всегда. Если сущности слишком разные (финансовый ledger + user profile + event log) — честно скажи «общий кирпич тут не уместен, оставляй раздельные сущности».

3. Заполни `главный-кирпич.md`:

```
# Главный кирпич

## Гипотеза
Описание кирпича с обязательными полями: id, тип, свойства.
Опционально (если применимо): родитель/дети (для иерархий), порядок (для упорядоченных коллекций), и т.п.

## Список вариантов
Каждая бывшая «отдельная сущность» становится кирпичом с типом='X' и своими полями в свойствах.

## Что я выигрываю
Какие операции теперь работают на все типы единообразно (поиск, история изменений, права доступа, экспорт, аудит, и т.п.).

## Что я проигрываю
Где этот кирпич неудобен — и почему я готов на это пойти.
```

Возможны три исхода:
1. **Один кирпич** — если унификация выигрышна для всех типов.
2. **Два кирпича** — если сущности кластеризуются в две группы.
3. **Без унификации** — если данные слишком разные, оставляй раздельные сущности.

Покажи варианты пользователю и спроси, что выбрать.

## Stage 5: 4 memory layers (AI-only)

**Conditional:** only run if the product has an AI agent or LLM component. If not — skip to Stage 6.

Read `context/architecture/01-обзор/` and `context/architecture/03-данные/главный-кирпич.md`.

Create `context/architecture/02-ядро/память-и-состояние.md` filling the 4-layer model:

```
# Где живёт память

## Слой 1. Сейчас (то, что в окне прямо сейчас)
- Что туда попадает на каждом запросе
- Лимит размера
- Что выкидывается первым

## Слой 2. Недавнее (что произошло за дни / часы)
- Где хранится (текстовые файлы с датами / отдельная таблица событий / git-папка памяти)
- Срок жизни
- Когда переезжает в Слой 3

## Слой 3. Постоянное (что я знаю про мир и пользователя)
- Где хранится
- Кто пишет
- Кто читает

## Слой 4. Навыки (опционально — если агент учится с опытом, накапливает процедурные паттерны)
- Где хранится
- Как обновляется

## Когда «недавнее» переезжает в «постоянное»
Например: раз в неделю ночной саб-агент читает события за 7 дней, извлекает повторяющиеся факты в постоянное, удаляет события старше 30 дней.

## Когда «постоянное» очищается
Например: если факт не подтверждался 90 дней — помечается «устарел», через 180 дней — удаляется.
```

Important:
1. Для маленьких проектов текстовые файлы + git побеждают векторные базы. Для узкого домена это экономит до 95% токенов. Векторную базу подключаем когда фактов больше 10 000.
2. Если процессы длинные — рекомендуй хранить память как git-репозиторий. Каждое изменение можно посмотреть и откатить.
3. Если проект маленький — честно скажи «вам это пока не нужно, начните с одного файла памяти».

If any data is missing — ask a clarifying question.

When done: show `память-и-состояние.md` and ask what to refine.

## Stage 6: triggers

Read `context/architecture/02-ядро/память-и-состояние.md` (if exists).

Create `context/architecture/04-потоки/` with `триггеры.md`.

Fill `триггеры.md` with a trigger table:

```
| Триггер | Тип | Что делает | Лимит времени | Защита от дублей | При сбое |
```

**Типы триггеров (КОГДА срабатывает):**
- действие юзера: нажал кнопку, отправил форму
- внешнее событие: кто-то прислал нам сигнал (платёж, входящее письмо, webhook)
- по расписанию: утром, ночью, раз в неделю
- внутреннее событие: юзер создал заказ → отправить письмо
- файловый сигнал: изменился файл (узкий случай — для AI-агентов на локальной машине)

**Режим выполнения (КАК выполняется):**
- синхронно: мгновенный ответ < 3 сек
- асинхронно (фоном): до часа
- ночью: до 8 ч

Для каждой строки таблицы заполни:
- Лимит времени
- Защита от дублей — если применимо (для триггеров «по расписанию» дубль = новый запуск, ок)
- При сбое: повторить / записать в журнал / уведомить меня

If AI agent — add `## Ночной режим (sleep-time)` section:

```
Что делает «ночной» саб-агент в 03:00:
- Читает события за день
- Извлекает факты в постоянную память
- Чистит дубли
- Готовит контекст к утру
- Лимит: 50 тыс. токенов / ночь
```

Important: для длинных процессов — каждый шаг отдельно, с повтором при сбое. Деплой не должен убивать 2-часовой процесс.

When done: show `триггеры.md` and ask what to refine.

## Stage 7: capabilities + permissions matrix

Read `context/architecture/01-обзор/`, `02-ядро/характер.md`, `04-потоки/триггеры.md`.

Create `context/architecture/02-ядро/способности/` subfolder (one file per capability) and `context/architecture/02-ядро/права-доступа.md`.

### Per-capability files

«Способность» — одно действие, которое продукт умеет делать. У AI-агента — каждый инструмент. У обычного продукта — каждое API-действие.

Структура (как онбординг для нового сотрудника):

```
# способность: <название>

## Когда применять
1-2 конкретных сценария.

## Когда НЕ применять
Где граница с другими способностями.

## Примеры
- Good: один пример правильного вызова
- Bad: один пример, когда вызывать НЕ нужно (и почему)

## Что может пойти не так
Что делать в случае ошибки.
```

### права-доступа.md

Колонки матрицы — роли пользователей, релевантные для проекта (например: Гость, Юзер, Админ, AI-автоматика; для подписочных — добавь Premium).

```
| Действие | <роль 1> | <роль 2> | ... | AI-автоматика |
```

Ячейки: `сам` / `спросить` / `эскалировать` / `запретить`.

**Правила:**
- Обычное и обратимое — сам
- Сообщения наружу, оплаты до X — спросить
- Деньги выше X, удаления, юр. действия — эскалировать (явное «да» от человека)
- То, что в `02-ядро/характер.md` помечено «Отказы» — запретить

Опционально: если хочешь дать пользователю настроить уровень автономии агента (от «делай всё сам» до «спрашивай по каждому шагу») — опиши, как это будет в UI.

When done: show tree of `context/architecture/02-ядро/` and ask what to refine.

### сущности.md

Опирайся на результат Stage 4:
- Если выбран «один кирпич» — все сущности наследуются от него (разные значения type)
- Если «два кирпича» — две группы наследования

```
| Сущность | Назначение | Основные поля | Связи |

### схема-связей.md

Mermaid ER-диаграмма в синтаксисе `erDiagram`. Пример:

```
erDiagram
    USER ||--o{ ORDER : places
    ORDER ||--|{ ITEM : contains
    ITEM }|--|| PRODUCT : refers
```

Все связи помечены: 1:1, 1:N, N:M.

### правила-нерушимые.md

```
# Правила, которые НИКОГДА не должны нарушаться

## Базовые
- Дата создания не может быть позже даты обновления
- Удалённые записи не возвращаются в обычных запросах

## Бизнес-правила (специфичные для проекта)
Например:
- Сумма заказа = сумма всех товаров
- Организацию нельзя удалить, пока есть активная подписка
- У каждого действия есть источник: юзер / система / агент
```

Important:
1. Каждое поле имеет причину. «Может пригодиться» — выкинь.
2. Каждая связь — через FK на id, не через значение поля (имя, email и т.п.).

## Stage 9: tech stack

Read `context/idea/06-principles/` and `context/architecture/03-данные/сущности.md` (+ `02-ядро/память-и-состояние.md` if AI).

### технологии.md

Колонки: `Слой | Выбор | Альтернативы | Почему именно это`. Заполни только строки для слоёв, релевантных проекту. Возможные слои:

- Язык
- Фреймворк
- База данных
- Хостинг
- Модель AI (только если есть AI)
- Очередь / расписание
- Email
- Платежи
- Аналитика
- (другое, что нужно проекту)

Если проект для конкретного локального рынка — учти местные требования: data residency, локальные платёжные провайдеры, налоговая регуляторика. **Регуляторика и список провайдеров меняются — не полагайся на память модели.** Сверься с актуальными источниками (официальные регуляторы, страницы провайдеров) и зафиксируй дату проверки в файле. Примеры (требуют верификации на момент сборки):
- **Россия (152-ФЗ):** база и продакшен-сервера в РФ. Хостинг: Яндекс.Облако / Selectel / Timeweb. Платежи: YooKassa / Yandex Pay / CloudPayments.
- **ЕС (GDPR):** хостинг в ЕС или сертифицированный SCC. Платежи: Stripe / Adyen / Mollie.
- Для других рынков — сам найди требования к data residency и подбери локальных провайдеров.

Если web-доступа нет — НЕ галлюцинируй регуляторику. Помечай `[требует проверки актуальности на YYYY-MM-DD]` и попроси пользователя верифицировать.

**Правило:** у стартапа 3 жетона на эксперименты. AI в продукте (LLM как компонент системы) — это уже один жетон. Остальное — самое скучное и проверенное:

- PostgreSQL > MongoDB
- Next.js / Rails / Django > новый фреймворк месяца
- Resend / Postmark > свой почтовый сервер
- Coolify / Railway > Kubernetes
- Stripe / YooKassa / локальный провайдер по рынку > свой биллинг

### что-не-выбрали.md

Что я сознательно НЕ выбрал и почему. Примеры:
- Микросервисы → решение для команд 200+ человек, не для нас
- Векторная база при <10 тысяч фактов → текстовые файлы экономят токены
- Свой LLM на сервере → аренда GPU дороже Anthropic API в 10 раз
- Сложные протоколы между сервисами → REST + JSON достаточно

Important: если предлагаешь экзотическое — объясни какой жетон тратится и почему оно того стоит. Если все три — отговори.

## Stage 10: premortem — 10 reasons it failed

**Dispatch as subagent.** Этот стейдж исполняется в изолированном контексте через Task tool (субагент). В основной сессии агент по инерции защищает то, что вы вместе написали; в свежей — беспристрастное зеркало. Не инструктируй пользователя открывать новый чат — оркестратор сам запускает субагента.

Read all of `context/architecture/`.

Create `context/architecture/07-нефункциональные/` with `риски-архитектуры.md`.

Act as if a year passed. Архитектура развалилась. Пользователь переписывает проект с нуля.

**Action:** назови 10 причин, почему это произошло. Не жалей. Будь жёстким. Не «возможно» / «вероятно». Говори как о факте, который уже случился.

Используй знания о реальных провалах. Применяй то, что релевантно типу проекта:

**Общие (применимо всегда):**
- Knight Capital: $440 млн за 45 минут из-за неудалённого мёртвого кода
- Healthcare.gov: $1.7 млрд из-за того, что не подумали об асинхронных очередях

**Специфичные для AI-агента:**
- Опасная тройка: личные данные + ненадёжное содержимое + отправка наружу = атака
- Накопленный мусор в памяти: продукт через полгода забыл, что он про
- Бесконтрольные расходы: в первый месяц на токенах сожгли месячную выручку

По каждой причине:
- Что пошло не так (1 предложение, в прошедшем времени)
- Почему именно у меня (ссылка на конкретный файл архитектуры)
- Что должен был сделать сейчас, чтобы избежать — и где это записать в архитектуре (конкретный файл/секция)

Распредели в `риски-архитектуры.md`:

```
## 3 убийственных риска
Без них продукт умрёт через 3 месяца.

## 4-5 серьёзных рисков
Снижают качество, но не убивают.

## Stage 11: security + costs

Read `context/architecture/02-ядро/права-доступа.md`, `02-ядро/способности/`, `04-потоки/триггеры.md` (+ `02-ядро/память-и-состояние.md` if AI).

Create two files in `context/architecture/07-нефункциональные/`: `безопасность.md` and `расходы.md`.

### безопасность.md

Базовый список:
- Где хранятся секреты (никогда в коде, только в env-переменных)
- Журнал действий: какие действия записываются с временем + причиной
- Личные данные: что храним, как защищаем
- Вход в систему: как юзер логинится, как живёт сессия (предпочитай magic-links/email-коды паролям)
- Проверка прав: как проверяем «можно ли»
- Лимиты на запросы: на какие эндпоинты, какие лимиты

**Если AI-агент — отдельный блок «Проверка опасной тройки»:**
- Есть доступ к личным данным? (мои файлы, база, ключи)
- Есть контакт с ненадёжным содержимым? (письма, веб-страницы, чужие чаты)
- Есть возможность отправить наружу? (email, telegram, API)

Если все три — описать пару агентов:
- **Читающий:** читает ненадёжное, БЕЗ инструментов, выдаёт структурированный результат
- **Действующий:** получает только структуру, имеет инструменты

Если хотя бы одну ногу отрезали — этого не нужно, объясни почему.

**Если b2b — добавь обязательный список:**
- Журнал действий
- Роли с разными правами (не просто «админ/юзер»)
- Вход через корпоративный аккаунт
- Экспорт данных
- Мягкое удаление + удаление по запросу пользователя
- Счета с НДС
- Уведомления внешним системам
- Лимиты на запросы
- Страница со статусом сервиса

### расходы.md

Таблица ресурсов:

```
| Ресурс | Цена | Объём в месяц | Итого |
```

Если есть AI — отдельная таблица распределения моделей:

```
| Задача | Модель | Макс токенов | Оптимизация (кэш / batch / etc.) |
```

**Правила оптимизации AI-расходов:**
- Дешёвая модель на простое — экономия 40-70%
- Кэш стабильной части промта — попадание ≥90%
- Пакетная обработка для не-срочного — минус 50%
- Дешёвая модель справляется с ~80% задач. Топовую (Opus и аналоги) включай точечно: глубокие исследования, бизнес-решения, разбор кода.

Important: считай реалистично. Если архитектура жжёт 50 тыс. рублей в месяц на 100 юзеров — это не бизнес, это благотворительность.

## Stage 12: roadmap

Read all of `context/architecture/`.

Create `context/architecture/08-дорожная-карта/` with `roadmap.md` — пошаговый план разработки от пустого репозитория до запуска. Каждый этап — checkbox-строка `[ ]` (не готов) / `[x]` (готов).

В верх файла добавь правило работы с архитектурой во время разработки в виде отдельного раздела:

```
## Правило: не переписывать архитектуру

Архитектура расширяется, не переписывается. Если по ходу реализации появляется новое архитектурное решение, создай папку `context/architecture/06-решения/журнал-решений/` (если ещё нет) и зафиксируй там запись с датой и причиной.
```

**Принципы построения roadmap:**
1. Этапы идут в порядке зависимостей (нельзя писать бизнес-логику до миграции БД).
2. Каждый этап — атомарный модуль, который можно зарелизить отдельно.
3. Каждый этап ссылается на конкретные разделы архитектуры — откуда брать контекст при реализации.

**Формат каждого этапа:**

```
- [ ] **N. <Краткое название>**
  - **Что делаем:** 1-2 предложения по существу
  - **Контекст из архитектуры:** список файлов/папок в `context/architecture/`, которые нужно прочитать при реализации
  - **Критерий готовности:** конкретный артефакт (миграция накатилась, деплой на staging работает, E2E-тест X проходит)
```

**Типовая последовательность (адаптируй под проект):**
1. Инициализация репозитория и стека (`05-стек/`)
2. Миграция БД (`03-данные/сущности.md` + `схема-связей.md`)
3. Базовый CRUD основных сущностей
4. Авторизация и матрица прав (`02-ядро/права-доступа.md`)
5. Первая фича — главная боль из `context/idea/04-mvp/`
6. Триггеры и фоновые задачи (`04-потоки/триггеры.md`)
7. AI-агент (если есть) — способности, память, опасная тройка
8. Интеграции (платежи, email)
9. Production hardening — мониторинг, security, лимиты расходов
10. Запуск

После создания `roadmap.md` пропиши на него жёсткие ссылки в двух местах, чтобы любой агент сразу его находил:

1. **`context/architecture/README.md`** — добавь видную секцию сверху:

   ```
   ## 🚀 Текущий план разработки
   [08-дорожная-карта/roadmap.md](08-дорожная-карта/roadmap.md) — пошаговый план с чекбоксами `[ ]` / `[x]`. Точка входа для разработки.
   ```

2. **`CLAUDE.md` в корне проекта** (создай, если ещё нет, или дополни существующий) — добавь секцию:

   ```
   ## Разработка
   План реализации: `context/architecture/08-дорожная-карта/roadmap.md`. Бери первый незакрытый этап `[ ]`, контекст подкачивай из указанных в нём файлов `context/architecture/`. После выполнения — меняй `[ ]` на `[x]`.
   ```

Important: не пиши код в этом стейдже — только roadmap и ссылки на него. Реализация пойдёт отдельным процессом по этому плану.

When done: show `roadmap.md` and updated cross-links in `context/architecture/README.md` and `CLAUDE.md`.
